# drugrep 設計原則（Design Principles）

## Principle 0（最上位原則）

> drugrep が保証するのは利用者の判断ではなく、判断材料である。ただし、その判断材料が誤った意味として解釈されないことについては、設計上の責任を負う。

---

## 既存原則の条文化

- 出典なしで発火するルールはゼロ。
- 出典の捏造禁止。
- Candidate は承認まで実装ファイルに入らない。
- golden 期待値の更新は1件ごと理由付き。

---

## Rule IV-1（Interpretation Principle）

- UI は内部モデルを医学的事実として表現してはならない。
- 評価モデル由来の区分・スケールは、その旨を必ず明示する。
- 判定基準: その語が持つ意味を**モデル自身が保証しているか**。保証していない意味を持つ語は**外部意味語彙**として扱う。

---

## Rule IV-2（Interpretation Validation）

- UI 変更時、実装者とは別のレビュー担当（現体制: Claude Code 実装 → Gin 実 UI 確認）が実 UI を読み、**利用者の解釈まで**検証する。
- レビュー単位は**文言ではなく画面状態**。
- 検証した画面状態の一覧を記録する。
- 最低確認:
  - 内部スケールが臨床事実として読めないか
  - 相対評価が絶対評価として読めないか
  - モデル区分が承認用量・治療域等と誤認されないか
  - 注意表示とスコア表示が矛盾して見えないか

---

## 5層品質保証体系 + User Decision

Contract → Data → Logic → Presentation → Interpretation → User Decision

| 層 | 名称 |
|---|---|
| L1 | Contract |
| L2 | Data |
| L3 | Logic |
| L4 | Presentation |
| L5 | Interpretation |
| — | User Decision |

注記: L1〜L5 は検証可能な層である。User Decision は検証対象ではなく、drugrep の責任がそこで正しく終わることの明示である。

## Track A 設計裁定（2026-07-07・Gin 承認 / Fable・チャッピー査読合意）

71mg Incident の裁定（`doseBands` は内部相対スケールで臨床上限ではない）の帰結として、以下を正式裁定とする。

- **裁定A（入力受理とモデル適用範囲の分離）**: 入力受理とモデル適用範囲は別責務。**入力できることは用量の妥当性を意味しない**。入力ゲートは機械的異常値のみ（`INPUT_SANITY_MAX`）で、`veryHighMax` を入力ゲートにしない。
- **裁定B（ロジックは計算・UI は非表示）**: `dose > veryHighMax` でも**ロジック層は数値スコアを計算する**。**UI は数値スコアを表示しない**（「適用範囲外」表示）。ロジックと表示の責務を分離する。
- **裁定C（判断材料の継続表示）**: 適用範囲外でも **タグ・相互作用・警告・救急導線（119）は継続表示**する。抑制するのはスコア数値のみ。
- **裁定D（複数薬の一括抑制）**: 複数薬入力で**1件でも適用範囲外なら集計スコア全体を非表示**にする。理由: モデル外の構成要素を含む合算値は数値として信頼できない。
- **裁定E（非表示理由の明示）**: UI がスコアを非表示にするときは**必ず理由を表示**する。黙って消さない（例:「〇〇がモデル適用範囲外のため、合算スコアは表示できません」）。

実装対応: `validation.ts`（A）/ `calculate-risk.ts`・`outOfModelRangeReason`（B・D の判定）/ `app/page.tsx`・`ScorePanel.tsx`（B・C・D・E）/ 構造化コード `ScoreSuppressionReason`（domain.ts）。ゴールデンで B・D のロジック挙動を固定済み。
