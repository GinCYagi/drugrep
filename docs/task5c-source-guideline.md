# SourceRef採用ガイドライン(Task5C 人間向け要件定義)

設計レビュー担当として作成します。前提: **AIは出典候補の提示までしか行わず、実在確認・内容一致確認・採用判断はすべて人間(Gin)が行う。** このガイドラインはその判断を標準化するためのものです。

---

## 1. 資料種別のランク付け

### Tier 1 — 積極採用(これだけでルール1件の根拠になり得る)

| 資料 | 理由・備考 |
|---|---|
| **添付文書(電子添文)** | 規制当局承認済みの一次情報。用量・警告・禁忌・相互作用の根拠として最優先。PMDA掲載の最新版を参照すること |
| **PMDA公表資料**(安全性情報、使用上の注意改訂指示、審査報告書) | 規制当局の公式判断そのもの |
| **インタビューフォーム(IF)** | 添付文書より詳細な薬物動態・薬理データ。日本病院薬剤師会様式に基づく製薬企業作成の公式資料 |
| **公的機関のガイドライン・資料**(厚労省、WHO、FDA、EMA等) | 特にFDA添付文書(label)は海外での警告根拠として有用 |
| **学会の診療ガイドライン**(日本ペインクリニック学会、日本臨床精神神経薬理学会等) | エビデンスレベル明記のものを優先 |
| **システマティックレビュー / メタ解析**(Cochrane等) | 査読論文の中で最上位 |

### Tier 2 — 条件付き採用(条件を満たせば採用可、可能ならTier 1と併用)

| 資料 | 採用条件 |
|---|---|
| **査読付き原著論文** | 掲載誌がPubMed/MEDLINE収載であること。単一の症例報告は「起こり得る」の根拠にはなるが「頻度・程度」の根拠にはしない |
| **査読付き総説(narrative review)** | 著者・掲載誌が確認でき、主張箇所に原典参照があること |
| **標準的教科書**(グッドマン・ギルマン、今日の治療薬、Martindale等) | 版と該当ページを特定できること。「教科書に書いてある」ではなく「第X版 p.Y」まで記録 |
| **中毒学の標準データベース**(TOXBASE、Poisindex等、アクセス可能な場合) | 引用時点の版・参照日を記録 |
| **公的な中毒情報センター資料**(日本中毒情報センター等) | 公開資料に限る |

### Tier 3 — 原則採用しない(例外は下記の限定条件のみ)

| 資料 | 例外条件 |
|---|---|
| **ハームリダクション系の非査読資料** | 一次情報(添付文書・論文)が存在しない経験的知見(乱用時の実態など)で、かつTier 1/2の根拠と矛盾しない補足としてのみ。単独根拠にはしない。採用時は `note` に「非査読・補足」と明記 |
| **プレプリント** | 原則不採用。査読通過を待つ |
| **製薬企業のプレスリリース・プロモーション資料** | 不採用(IFと添付文書で代替可能) |

## 2. 採用判断チェックリスト(1資料につき全問Yesで採用)

1. **実在性**: 自分でその資料の本体(PDF/書籍/公式ページ)を開いて確認したか。※AIが提示したタイトル・DOIをそのまま信用しない。DOI/URLが実際に解決することを確認
2. **一致性**: ルールが主張する内容(例: 「tramadol + moclobemide は相互作用リスク加算」)を、その資料の**該当箇所が直接**支持しているか。「関連しそう」ではなく「この記述がこのルールの根拠」と指させるか
3. **方向の一致**: 資料の結論とルールの方向が一致しているか(資料が「リスクは限定的」と言っているのに高リスクの根拠にしていないか)
4. **特定可能性**: 第三者が同じ資料の同じ箇所に到達できる情報(タイトル、発行元、年、版、URL/DOI、可能ならセクション/ページ)を記録したか
5. **鮮度**: 添付文書・ガイドラインは現行版か。改訂で内容が変わっていないか(参照日を記録)
6. **独立性**: 二次引用ではないか。「AがBを引用している」場合、Bそのものを確認したか
7. **利益相反の常識確認**: 単独根拠が製薬企業資料のみの場合、警告系(リスクを上げる方向)なら問題ないが、リスクを**下げる**方向の根拠なら独立資料を追加したか
8. **スコープ一致**: 資料の対象(用量域、投与経路、対象集団)がルールの適用範囲と一致しているか(治療用量の安全性データを過量域の根拠にしない、逆も同様)
9. **カタログ整合**: sources.ts の既存エントリと重複していないか。既存IDで足りるなら新規追加しない(`src(id, note)` の note で文脈を付ける)

## 3. 却下すべき資料(理由つき)

- **個人ブログ・まとめサイト・Q&Aサイト**(Yahoo!知恵袋等) — 検証不能
- **AI生成文章**(ChatGPT/Claudeの出力そのもの、AI生成の要約記事) — 出典ではなく出典探索の道具。**このプロジェクトではAI出力をSourceRefにすることを明示的に禁止**
- **Wikipedia** — 直接不採用。ただし脚注から一次資料を辿る入口としては有用(採用するのは辿った先)
- **二次引用のみ**(原典未確認の孫引き) — 伝言ゲームで内容が変質する
- **出典不明の医療系キュレーション記事** — 検証不能
- **薬物関連の掲示板・体験談**(5ch、Reddit等) — 検証不能。傾向把握の参考にしても出典にはしない
- **ニュース記事** — 一次資料(論文・当局発表)への案内としてのみ使い、採用するのは一次資料側
- **撤回された論文** — 参照前にRetraction Watch等で撤回確認(特に単独根拠の場合)
- **アクセスできない資料**(自分が本文を確認できなかったもの) — アブストラクトのみで採用しない。本文が確認できないなら別の資料を探す

## 4. ルール重大度ごとの必要十分条件

前提: リスクを**過大**に見せる誤りより、**過小**に見せる誤りの方が有害(ハームリダクションの非対称性)。したがって要求水準は「スコアを下げる/安心させる方向」ほど厳しくする。

| ルール種別 | 最低要件 | 推奨 |
|---|---|---|
| **high系ルール**(大幅加算、危険な相互作用。例: セロトニン症候群リスクのある併用) | Tier 1 × 1件(添付文書の警告・禁忌欄、またはPMDA/FDA警告) | Tier 1 + Tier 2 の計2件以上。相互作用ルールは「機序」と「臨床報告」の両面があると強い |
| **caution系ルール**(中程度の加算) | Tier 1 × 1件、または Tier 2 × 2件 | — |
| **info/低加算ルール** | Tier 2 × 1件以上 | — |
| **リスクを下げる方向の係数・ルール**(もし存在する場合) | Tier 1 × 1件 + 独立した資料1件の計2件。単一資料での「安全側への倒し」は禁止 | — |
| **base値・route係数・dose係数の根拠** | 各物質のbase/係数の設定根拠にも最低1件のSourceRef(添付文書の用量欄・IFの薬物動態データ等)。※現状ルール単位でしか出典を持てない設計なら、係数の根拠は該当物質の代表ルールまたはドキュメントに紐付け、Task5Cの保留事項として記録 | — |

**必要十分の「十分」側(過剰整備の防止):** 1ルールに4件以上の出典を付けない。件数より「チェックリスト全問Yes」の質を優先。MVPでは high系を最優先で整備し、全ルール網羅に時間がかかるならhigh → caution → infoの順で進める(ただしDoD上、**発火可能な全ルール**が最低要件を満たすまで完了にしない)。

## 5. 将来運用(物質・ルール追加時)

1. **順序の固定**: 「出典を先に確保してからルールを書く」。ルールを書いてから出典を探すと、確証バイアスで質の低い資料を採用しがち
2. **PRチェックリスト化**: ルール追加のコミット/PRテンプレートに §2 のチェックリストを含め、各項目にチェックを付けたものを残す(セルフレビューでも記録として残す)
3. **機械的ガード**: `assertRuleSources()` が「sources非空」を強制し続けること(Task5Cで実効性確認済みであること)。ガイドライン準拠の**質**は機械で担保できないので、質は人間、**量(非空)は機械**、と役割を分ける
4. **カタログの一元性維持**: 新規出典は必ず sources.ts に追加してから `src(id)` で参照。ルールファイルへの直書き禁止(既存方針の維持)
5. **鮮度の年次確認**: 年1回、Tier 1資料(添付文書・ガイドライン)の改訂有無を確認する軽量レビューを設ける。改訂で内容が変わっていたらルールと出典を同時に更新
6. **ChatGPT監査への組み込み**: クロスAIレビューの監査項目に「新規SourceRefのチェックリスト準拠」を追加。AIは実在確認はできないが、「ルールの主張と出典の要旨の論理的整合」の指摘には使える
7. **却下記録**: 検討したが却下した資料は、理由と共に軽く記録しておく(将来同じ資料を再検討する無駄を防ぐ)

---

## 6. Definition of Done(Task5C完了の客観条件)

以下すべてを満たした時点で完了:

1. **網羅**: 発火可能なすべてのルール(interactionRules含む)が、§4の重大度別最低要件を満たすSourceRefを持つ
2. **人間検証**: 全採用出典について§2チェックリストが実施済みで、その記録(チェック済みリスト、形式は問わない)が残っている
3. **カタログ整合**: すべての出典が sources.ts カタログ経由(`src()`)で参照され、カタログ外ID・直書きが存在しない
4. **機械的ガード**: `assertRuleSources()` が全ルール読み込み経路で実行されることが確認済み(sources空のルールを追加するとテストまたは起動時に失敗することを、意図的な失敗ケースで1回実証)
5. **テスト**: golden の全 ok:true ケースで sources非空アサーションが有効かつグリーン
6. **表示**: `/about` の出典一覧(`aggregateSources()`)に実データが表示されることを目視確認
7. **保留の明文化**: 出典が確保できず据え置いた項目(あれば)が、ルールの無効化または文書化された保留リストとして残っている。**「出典なしで発火するルール」がゼロ**であること
8. **監査**: ChatGPTによる「ルール主張と出典要旨の整合」レビューが完了し、指摘事項が解消または保留記録済み

---

補足を1点。§4の base/route/dose 係数の出典は、現行のデータ構造(ルール単位のSourceRef)では綺麗に持てない可能性があります。MVPではドキュメント紐付けで妥協し、「係数レベルの出典フィールド追加」はPost-MVP課題として docs/ideas か保留リストに落とすのが、リファクタリング禁止・MVP優先の方針と整合的です。
