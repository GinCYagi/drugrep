# 調査メモ: 処方薬取り込み機能（電子お薬手帳QR）

> ステータス: 現時点では実装しない（Post-MVP候補）
> 作成日: 2026-07-03
> 最終照合日: 2026-07-03（`src/lib/substances.ts` と突き合わせ）

## 1. 概要

薬局で受け取る調剤明細書等に印字される「電子お薬手帳QRコード」を読み取り、
処方薬情報を drugrep の入力（substanceId / 用量 / 経路）に変換して
リスク評価へ流し込む機能の実現可能性メモ。

## 2. 電子お薬手帳QRとは

- 一般社団法人 保健医療福祉情報システム工業会（JAHIS）が策定した
  「JAHIS電子版お薬手帳データフォーマット仕様書」に基づくQRコード。
- Ver.1.0 は2012年公開。薬局の調剤明細書等にQRとして印字され、
  スマホのカメラで読み取ると電子お薬手帳アプリに薬情報が自動登録される
  運用が普及している。
- 最新は Ver.2.6（2024年9月制定、JAHIS技術文書 24-104）。
  旧版 Ver.2.5（2024年3月、JAHIS標準 23-106）から更新された。
  既存アプリ（例: NOBORI）は Ver.1.0〜2.4 対応を明記しているものがあり、
  実運用では新旧バージョンが併存する。
- 仕様書はJAHISサイトで公開されており、引用明示のうえ自由に使用可能
  （改変時の準拠表明は禁止、という利用条件あり）。

## 3. JAHISフォーマットの構造（要点）

- CSV形式のレコード列。各行の先頭にレコード番号（レコードNo.）を持つ。
- 先頭にバージョンレコード（`JAHISTC` + バージョン番号。例: Ver.1.0 は
  `JAHISTC01`、Ver.1.1 は `JAHISTC02`）。
- 主な情報グループ:
  - 患者情報
  - 調剤情報（調剤年月日、調剤した薬局＝調剤－医療機関等レコード、
    調剤－医師・薬剤師レコード）
  - 処方情報（処方－医療機関レコード、処方－医師レコード）
  - 薬品レコード（No.201）: 薬品名・数量など
  - 用法レコード（No.301）: 用法および調剤数量（日数・回数等）
  - 薬品補足（No.281）・用法補足（No.311）: 省略可
- **RP番号**という処方指示単位があり、同じRP番号の薬品レコード群に対して
  同じRP番号の用法レコードが対応する（薬↔用法の紐付けはRP番号経由）。
- QR容量を超える場合はレコード単位でデータ分割され、
  分割制御レコード（No.911）が付与される。複数QRの読み取り・結合が必要になる。

### QRから取得できる主な項目（drugrep視点）

| 項目 | 取得可否 | 備考 |
|---|---|---|
| 薬品名（商品名） | ○ | 薬品レコード。基本は販売名（商品名）表記 |
| 薬品コード | ○（種別依存） | コード種別あり。**要確認**: レセプト電算コード / YJコードのどれが実運用で多いか |
| 1回量・1日量・数量 | △ | 用法レコード＋薬品レコードの組み合わせ。表記ゆれあり |
| 用法（服用タイミング） | ○ | 用法レコード。自由文の場合あり |
| 投与日数・回数 | ○ | 用法レコードの調剤数量 |
| 調剤年月日 | ○ | 調剤情報グループ |
| 投与経路 | △ | 剤形（錠・カプセル等）から推定は可能だが明示フィールドは**要確認** |

## 4. drugrepで取り込む場合に必要な変換

### 4.1 商品名 → 一般名

- QRの薬品名は商品名（販売名）が基本。現行マスターで解決できる例:
  「リリカ」→ `pregabalin`、「ルネスタ」→ `eszopiclone`、
  「トラムセット」→ `tramadol_combo`（トラマドール/アセトアミノフェン配合剤）。
  ※ 単剤トラマドール（「トラマール」等）は配合剤とは別実体で **現行マスター未対応**
    （`substances.ts` に alias を含めない方針）。取り込んでも §4.2 の「未対応物質」側に回る。
- drugrep の `substances.ts` には既に日本語・商品名エイリアスがあるため、
  **既存のエイリアス解決機構をそのまま再利用できる**のが大きい。
  解決の実体は `src/lib/find-substance.ts` の `findSubstance()`。ただし
  **trim + lowercase の完全一致のみ**（部分一致・スペルミス吸収なし）。
- したがってQRの表記「規格込み」（例: 「トラムセットOD錠」）は正規化なしでは
  絶対にヒットしないため、規格・剤形サフィックスを剥がす正規化レイヤーが必要。
- 薬品コード（YJコード等）が取れる場合はコード→一般名の対応表の方が堅牢。
  **要確認**: コード種別ごとの出現頻度と、対応表の入手性・ライセンス。

### 4.2 一般名 → substanceId

- 現行5物質（`methylphenidate`, `moclobemide`, `pregabalin`, `tramadol_combo`,
  `eszopiclone`）にヒットしたものだけ取り込み、それ以外は「未対応物質」として明示スキップ。
- 未対応物質を黙って捨てると相互作用評価が欠落するため、
  「読み取ったが評価対象外の薬がある」ことをUI上で必ず表示すべき
  （ハームリダクション的に重要。評価の網羅性を誤認させない）。

### 4.3 用量・用法・日数

- 「25mg錠 × 3錠/日 × 14日」のような構造から、
  drugrep の入力（1回量 or 1日量）への変換が必要。
- **単位系が物質ごとに異なる点に注意**: mg 基準の物質（methylphenidate,
  moclobemide, pregabalin, eszopiclone）と、**錠数基準の `tramadol_combo`**
  （`defaultUnit: "tablet"`、doseBands も 8/12/16 錠）が混在する。
  配合剤は「37.5mg × N錠」を mg に潰さず錠数へ変換する分岐が要る。
- 規格（mg/錠）× 個数 の掛け算が必要になるケースと、
  mg直接表記のケースが混在する想定。**要確認**: 実データでの表記パターン。
- 用法が自由文（「疼痛時 頓用」等）の場合のパース戦略は未定。
  MVPの評価モデルは単回用量ベースなので、頓用・分割服用の解釈方針も未定。
- 投与経路: 処方薬は基本 oral と仮定してよいか。**要確認**:
  貼付剤・注射剤が混ざるケースの扱い（現行5物質では実質 oral が支配的だが、
  `methylphenidate` の routes は `["oral", "rectal", "nasal"]` で oral 限定ではない）。

## 5. MVPではやらない理由（Post-MVPに回す判断根拠）

1. **入力パースの複雑さが評価ロジックと無関係**
   QR読み取り・分割QR結合・CSVパース・表記ゆれ正規化はすべて
   入力層の問題であり、コアのリスク評価（Task4でゴールデンテスト固定中）
   に価値を足さない。MVPの安定化を優先する方針と競合する。
2. **バージョン分岐コスト**
   Ver.1.0〜2.6 が併存し、薬局側の対応バージョンもばらばら。
   互換レイヤーの維持コストが大きい。
3. **カメラ/QR読み取りはWeb MVPのスコープ外**
   モバイル前提の機能であり、計画中のReact Native/Expo展開と
   セットで設計する方が自然。
4. **医療情報の取り扱い**
   患者氏名・医療機関名など個人情報を含むデータを読むことになり、
   保存しない設計（読み取り即破棄）を含めたプライバシー設計が先に必要。
5. **失敗時の安全性**
   用量の誤パースは誤ったリスクスコアに直結する。
   確認UI（パース結果をユーザーが修正してから評価）が必須で、UI工数が大きい。

- [ ] 投与経路 or 剤形を示す明示フィールドの有無
- [ ] 1回量/1日量/数量の実データでの表記パターン（実QRサンプル入手）

- [ ] 商品名正規化（規格・剤形サフィックス除去）のテストケース収集
- [ ] マイナポータル経由の調剤情報取得APIとの比較（QRより筋が良い可能性）
- [ ] 単剤トラマドールを `substances.ts` に追加するか（未対応のままだと配合剤しか
      取り込めず、本機能の実用価値が下がる。マスター側TODOと連動）

## 7. 結論

**MVPでは対象外、Post-MVP候補。**
既存のエイリアス解決と `substanceId` 設計がそのまま土台になるため、
アーキテクチャ変更なしで後付け可能。着手時は「QRパーサ（入力層）」を
`src/lib/` 配下の独立モジュールとして追加し、evaluate 層には触れない構成を推奨。
